ProPerGUI: Benchmarking Proactive and Personalized GUI Agents

WWW 2026 Anonymous Submission

Abstract

Graphical User Interface (GUI) agents have emerged as a prominent research topic, attracting widespread attention. However, current research is largely confined to a reactive paradigm and often neglects user personalization, which fundamentally constrains the real-world utility of these agents. To address these limitations, we introduce a high-quality benchmark, ProPerGUI, for Proactive and Personalized GUI agents, spanning diverse platforms and application domains. We formulate this challenge through two core subtasks: Proactive Preference-Aware Intention Interpreting and Preference-Conditioned Execution, supported by a novel, unified evaluation framework. Comprehensive experiments across leading MLLMs and GUI Agents demonstrate that our benchmark poses substantial challenges, revealing significant limitations in proactive reasoning and preference alignment. Moreover, fine-tuning considerably can enhance the model's ability to understand user intentions and preferences. This work establishes a solid research foundation for proactive and personalized GUI agents, facilitating the development of the next generation of human-aligned GUI agents.

Task Definition

We design the task with two consecutive subtasks, as shown in Figure 1. To rigorously evaluate proactive and preference-aware GUI agents, we introduce a two-stage framework corresponding to these two complementary subtasks. The first, Proactive Preference-Aware Intention Interpreting, focuses on inferring user intentions, operational preferences, and the need for proactive assistance from historical interactions. The second, Preference-Conditioned Execution, evaluates whether the agent can generate executable actions aligned with the inferred intentions and preferences under the current GUI state. This design captures the dual requirements of proactive agents: first, understanding what the user aims to do and how they prefer to operate, then translating that understanding into concrete actions. By decoupling reasoning from execution, the framework supports fine-grained evaluation of both cognitive modeling and operational performance.

Subtask I: Proactive Preference-Aware Intention Interpreting.

The agent receives a historical interaction sequence $\mathcal{H} = \{(a_t, s_t)\}_{t=1}^{T}$, where $a_t$ denotes a user action (e.g., click, type) and $s_t$ the corresponding contextual information. The agent then generates a structured output $\hat{X} = \{\hat{I}, \hat{P}, \hat{D}\}$, which contains three key elements: Task Intention ($\hat{I}$), describing the user-agnostic goal the user is pursuing; User Preference ($\hat{P}$), describing behavioral tendencies or interaction styles (e.g., shortcut usage); and Proactive-Help Decision ($\hat{D}\in\{0,1\}$), a binary indicator of whether the agent should proactively assist. This subtask focuses purely on semantic and cognitive understanding of user behavior, forming the foundation for subsequent action generation in Subtask II.

Subtask II: Preference-Conditioned Execution.

This subtask evaluates whether the agent can effectively translate inferred cognitive understanding—captured through $(\hat{I},\hat{P})$—into precise and executable GUI actions. Specifically, given the same inferred intention and preference representation and the current GUI state $s_{T+1}$, the agent must produce the executable sequence $\hat{A}=\{\hat{a}_1,\hat{a}_2,\ldots,\hat{a}_K\}$ that both completes the task and remains aligned with the user’s operational preferences, ensuring consistency between reasoning and execution.

Dataset Overview and Statistical Analysis

As shown in Figure 2 and Figure 3, our benchmark comprises 1,503 instances across four platforms (Windows, Ubuntu, macOS, and Android), covering 34 applications and nine domains: Creativity, Browser, Education, Entertainment, Office, Programming, Productivity, Shopping, and Social. Each task contains an average of 20 interaction steps, with sequence lengths ranging from 4 to 64. Figure 3 (top) further shows that the average step counts across domains are relatively balanced, indicating good coverage and diversity.

We also analyzed the step-length distributions of all tasks, as well as those of Subtask I and Subtask II. Subtask I exhibits longer interaction sequences than Subtask II, as it focuses on extracting finer-grained user intentions and operational preferences. This design is essential for proactive GUI agents, enabling them to better infer user goals and deliver personalized, context-aware assistance. To preserve dataset authenticity and reliability, we ensured diverse step-length distributions across both subtasks.

The dataset was divided by platform, with 813 training and 240 validation instances on Desktop, and 346 training and 104 validation instances on Mobile. Splits were performed according to task-length distributions within each application to preserve domain balance and comparable complexity across sets. Definitions of the Desktop and Mobile action spaces are provided in the Appendix.

Appendix: More Prompt Examples

Subtask I: LLM-based Validation 

You are a GUI Behavior Analysis evaluation-only judge. Output ONE JSON object and nothing else.
Do NOT reveal chain-of-thought. Do NOT invent facts or use external knowledge. Do NOT add text outside JSON.

[GOLD_INTENT]   // one sentence
{GOLD_INTENT_DESCRIPTION}

[PRED_INTENT]   // one sentence
{PRED_INTENT_DESCRIPTION}

[JUDGMENT SCOPE]
- Judge ONLY whether GOLD_INTENT and PRED_INTENT express the SAME goal.
- Only the semantic goal expressed by the two sentences. Ignore wording style, synonyms, and implementation details.

[DECISION RULES]
- Output "match" ∈ {1, 0}.
  - 1 (correct): goals align in object + operation + scope; no broadening/narrowing; differences are paraphrase only.
  - 0 (incorrect): in all other cases, e.g.:
    * scope/granularity mismatch (one contains the other; missing/extra key sub-goal or constraint);
    * different or conflicting objects/operations;
    * upstream vs downstream stage mismatch (e.g., find/locate vs purchase/submit/confirm);
    * negation/contrast flips the goal (e.g., cancel vs submit);
    * sentence is empty;

[CONSTRAINTS]
- Do not infer unstated entities/steps/constraints from world knowledge.
- Do not output any evidence list; grounding is internal-only.
- If a sentence bundles multiple sub-goals and the other states one of them → usually 0

[STRICT OUTPUT JSON ONLY]
{
  "match": 1 | 0,                  // 1=correct (same goal), 0=incorrect
  "reason": "<=20 chars; only claims supported by cited actions>"
}

[VALIDATION]
- Do not output code fences or extra text. Only the JSON object above.

Subtask II: Preference-Conditioned Execution Prompt (Windows) 

You are a GUI action prediction assistant for Desktop (Windows).
You are given user action history, intention and preference, with a screenshot.
You need to predict exactly ONE next user action to perform ON THIS SCREEN.
- Metadata Description: 

  [History_JSONL]: history of user's actions

  [Intention]: user's intention of whole task

  [Preference]: user's preference of whole task

Output Format
Thought: ...
Action: ...

Action Space:
click(point='x1 y1')
left_double(point='x1 y1')
right_single(point='x1 y1')
drag(start_point='x1 y1', end_point='x2 y2')
hotkey(key='ctrl c')  # split keys with a space, lowercase, ≤4 keys
type(content='xxx')   # escape \', \" and \n; submit with trailing \n
scroll(point='x1 y1', direction='down or up or right or left')
wait()                # sleep 5s and take a screenshot to check for any changes.
finished(content='xxx')  # escape \', \" and \n

Note
- Use {language} in Thought.
- Write a small plan and finally summarize your next action (with its target element) in ONE sentence in Thought.
- Prioritize actions that align with the user's preferences whenever possible.
- If the input history operation element is null, it means the element was not found at that time. The history element description may be inaccurate sometimes.
